Level 1 CPU — Architecture & FSM Walkthrough

This is a multi-cycle, single-bus, 16-bit micro-architecture. All data movement happens over one shared 16-bit bus. Every cycle, the controller asserts exactly one source onto the bus (via tri-state) and zero or more destinations latch from it. Multi-operand operations (ADD, SUB) need multiple cycles because the bus can only carry one value at a time.

Module hierarchy

cpu_level1_TB             ← drives test instructions, checks register state
└── cpu_level1            ← top: wires all the units to the shared bus
    ├── register_file     ← 8x R0..R7, each with its own tri-state buffer
    │   └── register × 8       parameterized 16-bit reg (enable + sync reset)
    ├── register          ← A   (operand 1 for ALU, written from bus)
    ├── alu                  ← combinational: ADD or SUB on A and bus
    ├── register          ← G   (latches ALU result, drives bus via G_tri)
    ├── register          ← Status (latches Z/N flags from ALU)
    └── controller        ← FSM: decodes opcode, sequences T-states, drives all
                              reg_en[7:0], reg_tri[7:0], A_en, G_en, status_en,
                              g_tri, imm_tri, alu_op, done

Controller FSM

The controller is a simple step-counter FSM. step goes from 0 → 1 → 2, but the controller asserts done and resets step to 0 once the current opcode's last cycle is reached. So a 1-cycle instruction (LDI, MOV) just stays at T0; a 3-cycle instruction (ADD, SUB) walks T0 → T1 → T2.

What happens in each T-state

Instruction T0 T1 T2
LDI Rd, K imm_tri=1, reg_en[Rd]=1 → Rd ← sign_ext(K) — —
MOV Rd, Rs reg_tri[Rs]=1, reg_en[Rd]=1 → Rd ← Rs — —
ADD Rd, Rs reg_tri[Rd]=1, A_en=1 → A ← Rd reg_tri[Rs]=1, alu_op=ADD, G_en=1, status_en=1 → G ← A + Rs g_tri=1, reg_en[Rd]=1 → Rd ← G
SUB Rd, Rs same as ADD: A ← Rd same shape, alu_op=SUB → G ← A − Rs same: Rd ← G
Why does ADD take 3 cycles? The ALU needs two operands (A and bus), but the bus can only deliver one value per cycle. So T0 grabs Rd into A, T1 puts Rs on the bus and the ALU adds (A + bus = Rd + Rs), T2 routes G back through the bus into Rd. Each T-state asserts a unique driver onto the bus.

Instruction encoding

LDI:        [op 4][Rd 3][imm 9]
MOV/ADD/SUB:[op 4][Rd 3][Rs 3][unused 6]

opcodes:
  0001 LDI   Rd, K9        Rd ← sign_ext(K9)
  0010 MOV   Rd, Rs        Rd ← Rs
  0011 ADD   Rd, Rs        Rd ← Rd + Rs
  0100 SUB   Rd, Rs        Rd ← Rd − Rs
